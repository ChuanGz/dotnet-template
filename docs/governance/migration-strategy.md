# Template Migration Strategy

## Purpose

Enable projects generated by `dotnet-service` to adopt later template decisions without overwriting product-specific code or requiring repository authors to understand each consuming system.

## Migration contract

Generated projects must record the template identity, template release, option-contract version, and resolved option values in a machine-readable metadata file. Consumers own their generated projects after creation; a template release never implies automatic source replacement.

Each release must classify generated-output changes as:

| Change | Required migration support |
|---|---|
| Internal fix with no contract impact | Changelog and verification command |
| Additive generated capability | Opt-in instructions and rollback |
| Changed default for new projects | Before/after behavior and existing-project impact |
| Changed generated contract | Step-by-step migration, compatibility impact, and validation |
| Removed option or behavior | Completed deprecation process and replacement path |

## Migration deliverable

A release requiring consumer action must provide:

- affected template and contract versions;
- applicable option combinations;
- prerequisites and backup boundary;
- ordered changes with expected intermediate state;
- build, test, security, and operational validation;
- rollback or recovery procedure;
- known non-automated decisions;
- support and escalation path.

Automation may assist mechanical changes, but it must support dry-run behavior, report every changed file, avoid product-owned business code, and be safe to rerun. A script is not a substitute for migration documentation.

## Compatibility rules

- Patch releases must not require consumer source migration.
- Minor releases may add opt-in behavior but must preserve existing stable defaults.
- Stable breaking changes require a major release.
- Before `1.0.0`, breaking changes remain possible but still require explicit impact, migration, and changelog evidence.
- Security or unsupported-runtime remediation may accelerate change; the release must document why normal compatibility cannot be preserved.

## Acceptance evidence

- A project can identify its originating contract and resolved options.
- Migration scope is determined without guessing from folder structure.
- Validation proves the project still builds and its selected capabilities still work.
- Product-specific changes remain owned by the consuming team.
